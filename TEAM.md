# PayIncus / Incudal — 固定开发团队章程（TEAM.md）

> 本文件定义本项目**固定的虚拟开发团队**与**任务分诊协议**。
> **规矩**：以后处理**任何**任务,主程(Tech Lead)都要**先做角色分诊**——判断这个任务需要哪些角色、按什么顺序、谁执行——再派活。不要一上来就动手。
> 与 `AGENTS.md`(统一工作规矩)、`OPERATIONS_HANDOFF.md`/`HANDOFF.md`(运维/版本)配套使用。

## 0. 主程 / Tech Lead（= 我,不可外派）
定根因、切任务、**角色分诊**、写/审 spec、守不变量、跨角色集成、去重、对抗式复核、跑守卫、真机验证、管 git（**仅 owner 明确同意才 commit/push/OTA**）。所有产出最终由主程集成 + 验收。

## 2. 任务分诊协议（每个任务必走）

1. **接单 → 角色分诊**:主程先判断——这个任务触及哪些领域?需要上面哪几个角色?顺序?串行还是并行?
2. **高危先 spec**:凡触及 `AGENTS.md §2` 高危路径(鉴权/支付/权限/风控/资源交付/兑换/实例创建/OTA)——技术架构师先读代码+守卫、出 spec,再交开发执行。
3. **派活**:
   - 🔵 Claude 角色 → 用 `Agent(subagent_type: "<role-file>")` 调用(见 `.claude/agents/`)。
   - 🟢 codex 角色 → `codex exec -s <sandbox> - < 角色preamble+任务` 调用(preamble 见 §3;审计/规划用 `read-only`,改代码用 `workspace-write`)。
4. **串并行**:改**共享文件**(schema.prisma/app.ts/package.json/locales/router 等)的任务**必须串行**;彼此独立文件可并行。只读分析/测绘可放心并行。
5. **验收闭环**:开发完 → QA 工程师跑相关守卫 + 对抗式复核 → 主程集成、真机验证、报告。**任一守卫不过 = 未完成。**
6. **git/发版**:一律 owner 明确点头才 commit/push/OTA(硬规矩)。

## 3. codex 角色调用 preamble（🟢 角色统一前缀）

每个 codex 角色任务,prompt 顶部加该角色声明,再接主程给的 spec/清单:
> 你是 PayIncus/incudal 项目的**<角色名>**。遵守 `AGENTS.md` 全部规矩:高危路径先读后写、改动最小化、每次改动过对应 `test-*-guards.ts`、锁定表不改横向滚动、不泄密、不擅自 commit/push/OTA。产出后自跑相关守卫 + type-check,全绿才算完,并报告改了什么/为什么/怎么验证的。
>
> **编码规范(Karpathy 四准则,必守)**:①**先想后写**——显式写出假设,有更简解法就说,模糊就停下问,别硬猜。②**简单优先**——只写解决问题的最小代码;不加没要求的功能/抽象/"灵活性"/对不可能场景的错误处理;200 行能压 50 就压。③**外科式改动**——只碰必须碰的;不顺手"改进"无关代码/格式/注释;匹配既有风格;只清理你自己改动造成的孤儿 import/变量,不删既有 dead code(发现了只提不删);每一行改动都要能直接追溯到需求。④**目标驱动**——把任务变成可验证目标(写测试/守卫再让它过);多步先列"步骤→验证"计划。

沙箱:审计/测绘/复核 = `-s read-only`;实现/修改 = `-s workspace-write`。模型默认 gpt-5.6-sol。

## 4. 常见任务的分诊示例

- **改一个业务 bug(高危面)**：架构师(定根因+spec)→ 后端/前端开发(codex 实现)→ QA(守卫+对抗验证)→ 主程集成验收。
- **新增一个功能**：产品经理(需求/验收)→ 业务分析师(影响面)→ 架构师(spec)→ 后端+前端(codex 并行/串行)→ UI(codex 样式)→ QA → 主程。
- **UI 改版**：UX(方向/token)→ UI(codex 落地)→ 移动端(codex 响应式)→ QA(布局/i18n/a11y 守卫)→ 主程双端 build。
- **全量审计**：架构师+QA(Claude 多透镜)+ 后端/前端(codex 独立复核)并行只读 → 对抗式互验 → 主程汇总分级报告。
- **部署/发版**：运维(codex 脚本/校验)→ 主程真机验证 → owner 批准才 OTA。
