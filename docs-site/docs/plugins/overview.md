# 扩展中心

PayIncus 扩展中心用于安装和管理第三方扩展。扩展可以提供后台配置页，也可以通过受控客户端扩展点向用户端增加入口或页面。

完整平台化路线见：[扩展中心方案](/plugins/platform-plan)。实时在线目录见：[扩展市场](/plugins/market)。

扩展不会直接修改 PayIncus 源码。安装后的文件保存在运行态目录：

```text
/opt/incudal/plugins
/opt/incudal/plugin-data
/opt/incudal/plugin-logs
/opt/incudal/plugin-staging
```

这些目录会在 OTA 更新和回滚时保留。

## 能力

- 上传 `.tar.gz` 扩展包安装
- 从带治理字段的稳定在线扩展市场安装
- 启用、禁用和卸载扩展
- 查看安装任务日志
- 安装后在后台左侧菜单显示扩展设置入口
- 通过独立扩展设置页管理业务配置
- 通过 sandbox iframe 加载第三方后台或用户端扩展页面
- 通过 manifest 声明 webhook action、事件订阅、通知模板、服务扩展、支付网关扩展和插件存储
- 通过 OAuth Provider、API Token、OpenAPI 3.1 和 Public API SDK 开发第三方服务端集成
- 通过主题包市场、上传、审核、预览、启用、回滚和配置表单接入受控主题
- 使用内置模板开发扩展，例如 `plugin-templates/ai-ticket-agent-plugin` 提供 AI 工单草稿扩展模板

## 平台化目标

当前扩展中心已经具备第三方开发平台首版能力：它可以安装和治理扩展包，渲染受控的后台页、用户页和扩展配置页，也提供 Public API、OAuth Provider、SDK、webhook action、事件、插件存储和主题系统。它仍然不会把任意后端代码执行、任意数据库写入、支付资金状态或资源交付状态直接交给扩展。

PayIncus 扩展平台 2.0 的目标是让第三方开发者可以在受控边界内开发完整功能：

- 通过 `/api/v1` 和 OpenAPI 3.1 使用稳定公共 API。
- 通过 API token 和 scope 获得明确授权。
- 通过 OAuth Provider 创建第三方应用、授权码换 token，并用 `poa_` Bearer token 访问公共 API。
- 通过资源 API 读取产品目录、当前用户服务、当前用户订单和当前用户工单，并受控排队服务启动/停止/重启任务、创建工单和追加工单公开回复。
- 通过扩展 action runtime 执行受控后端动作。
- 通过事件订阅响应订单、支付、实例、工单和通知事件。
- 通过扩展私有存储保存业务数据。
- 通过主题包安装、预览、启用、回滚和配置表单加载受控主题 CSS。

这套目标会优先保护认证、支付、权限、风控、资源交付和审计链路。第三方能力应通过平台 SDK/API 进入业务系统，而不是绕过 PayIncus 的内部校验。

## OAuth Provider

PayIncus 已提供 OAuth Provider 首版。管理员可以在 OAuth 设置页创建对外 OAuth App，第三方应用通过浏览器 `/oauth/authorize` 授权确认页获取 authorization code，再用 code 换取 `poa_` access token 和 `por_` refresh token，并按 scope 调用 `/api/v1`。当前首版已经覆盖 client secret hash 存储、redirect URI 精确匹配、scope 校验、token 过期、refresh token 轮换、撤销和授权记录管理。

## 公共资源 API

PayIncus 已提供 `/api/v1` 资源 API 首版。第三方服务端可以用 `pat_` API Token 或 `poa_` OAuth access token 访问 `/api/v1/me`、`/api/v1/balance`、`/api/v1/balance/logs`、`/api/v1/balance/adjustment-requests`、`/api/v1/billing-records`、`/api/v1/products`、`/api/v1/services`、`/api/v1/orders`、`/api/v1/tickets`、`/api/v1/notifications` 和 `/api/v1/plugins` 相关公开资源。`profile:write` 只允许更新 `avatarStyle`；`balance:write` 只允许提交当前 token 用户自己的待审批余额调整申请，不直接写余额、流水、支付或充值订单；`services:operate` 只允许给当前 token 用户自己的服务排队启动、停止或重启任务，再轮询公开任务状态，或取消仍在等待中的公开电源任务；`services:billing` 只允许续费当前 token 用户自己的付费服务，续费会复用内部余额锁、扣款、账单和托管收入状态机；`tickets:write` 只允许创建自己的公开工单、公开回复、受控图片附件、关闭或重新打开自己的工单；`notifications:send` 只允许给当前 token 用户自己发送短文本通知、平台白名单模板或已启用扩展声明的受控通知模板；`plugins:action` 只允许发现和触发已启用扩展声明过的公开 webhook action。

服务、订单、账单、工单、通知和扩展 action 接口只返回当前 token 用户或已启用扩展的受控数据，不暴露 root 密码、Incus ID、宿主机内部配置、支付回调、provider 配置快照、内部备注、原始存储文件 ID、webhook URL、secret 或扩展配置值。服务操作首版不开放创建、暂停、恢复、重装、删除、迁移或宿主机操作，支付建单、支付回调、退款、直接余额写入、敏感用户资料、扩展安装启停和主题启用也不通过 Public API 开放。

## 主题系统目标

真正的主题系统独立于普通页面扩展。主题包需要声明兼容版本、设计 token、布局区块、静态资源、配置表单和覆盖范围。当前首版已经支持后台上传主题、从稳定在线主题市场安装主题、主题投稿审核/扫描/发布、预览主题、启用主题、回滚到默认主题、保存主题配置，并通过 `/api/themes/active` 让用户端加载当前主题 CSS 和配置值。

主题系统允许覆盖视觉和受控布局区块，不允许注入未授权远程脚本、绕过登录/支付/权限页面逻辑，或修改后端业务规则。

## 后台界面

扩展中心在后台内部分为三个页面：

- 已安装：上传扩展包、查看扩展列表、启用/禁用/卸载扩展和打开扩展设置。
- 扩展市场：单独展示稳定在线市场索引，管理员可以刷新市场并安装市场扩展。
- 安装任务：查看上传安装、市场安装、启用、禁用和卸载任务；任务列表固定每页最多 7 条，更多任务通过分页查看，右侧日志面板显示所选任务输出。

扩展中心只负责安装、启用、禁用、卸载和查看任务。扩展安装后，如果 manifest 声明了 `admin.plugins.settings` 后台设置页，PayIncus 会把扩展设置入口加入后台左侧菜单；未启用扩展也可以先进入设置页维护配置，页面会提示当前启用状态。设置页面使用独立路由打开，不再放在扩展中心右侧详情栏里。

## 扩展市场治理

扩展市场索引默认托管在文档站稳定目录，也可以配置到受信 CDN 或 GitHub Release 目录；每个扩展必须声明治理信息。后台默认只展示 `reviewStatus = listed` 的扩展；`pending`、`delisted` 和 `rejected` 扩展不会进入默认市场列表，也不能通过安装接口绕过安装。

市场页会显示：

- 审核状态：待审核、已上架、已下架、已拒绝
- 可信来源：官方来源、认证开发者、第三方
- 开发者资料：名称、主页、GitHub、认证状态和联系信息
- 版本兼容：最低和最高 PayIncus 版本
- 权限声明：后台页面、用户端页面、API 能力和存储目录
- 安全信息：SHA256 固定校验、签名状态、索引指纹
- 商业预留：免费/付费、价格、币种、分成比例
- 运营指标：评分、评分数量、安装量、升级说明和回滚说明

安装市场扩展前，后台会弹出安全确认，列出来源、审核状态、兼容范围、SHA256 和权限声明。服务端会再次校验 GitHub Release 下载地址、SHA256、审核状态和版本兼容范围；这些校验不依赖浏览器端。

## 第三方上传与审核

第三方扩展有两条不同入口：

- 私有上传安装：超级管理员可以在后台上传 `.tar.gz` 包，包会先进入扩展 staging 目录，服务端校验 manifest、路径安全、入口文件和 SHA256 后再安装。这条通道适合自用、内测和线下交付，不等同于进入公共扩展市场。
- 市场投稿审核：第三方开发者提交扩展包、源码仓库、Release artifact、manifest、权限说明、兼容范围、SHA256、截图、更新说明、回滚说明和联系方式。候选包不能直接出现在用户市场，也不能被普通管理员绕过审核安装。

推荐市场投稿流程：

1. 开发者在自己的 GitHub 仓库发布 Release，并上传扩展 `.tar.gz` artifact。
2. 开发者提交审核资料：仓库地址、Release 下载地址、manifest 地址、SHA256、开发者资料、权限声明、兼容范围、定价信息和安全说明。
3. 审核系统把条目标记为 `pending`，只进入审核队列，不进入默认市场列表。
4. 审核人员执行自动校验：下载地址必须是 GitHub Release artifact，SHA256 必须匹配，manifest 必须合法，包内不能有绝对路径、`../`、软链接、硬链接，入口文件必须存在。
5. 审核人员执行人工校验：权限是否最小化、是否触碰支付/订单/资源交付/认证/风控高风险能力、文档是否完整、截图和功能说明是否一致、回滚说明是否可执行。
6. 通过后把 `reviewStatus` 改为 `listed`，并把固定 SHA256、版本、兼容范围和权限声明写入市场索引。
7. 拒绝、下架或发现漏洞时使用 `rejected` 或 `delisted`，默认市场不展示，安装接口也不能安装。

审核记录应至少保存：开发者、扩展 ID、版本、提交时间、审核人、审核结论、SHA256、权限快照、兼容范围、风险等级、拒绝/下架原因和复审记录。

正式市场安装只信任审核后的市场索引和固定 SHA256。即使开发者重新上传同名 Release artifact，只要 SHA256 不一致，服务端也会拒绝安装。

## AI 工单扩展模板

`plugin-templates/ai-ticket-agent-plugin` 是官方扩展模板，用于第三方开发 AI 工单辅助扩展。模板要求 PayIncus `v0.3.2` 或更高版本。

官方市场会提供 `AI 工单助手` 扩展包。生产环境默认读取 `https://payincus.com/plugin-market/index.json`，管理员进入后台扩展中心的“扩展市场”页刷新并安装。安装完成后，后台左侧菜单会显示 `AI 工单助手` 设置入口；需要实际生成草稿或接管回复时，再在“已安装”页启用扩展。

`AI 工单助手` 使用独立中文设置页，不需要编辑配置 JSON。设置页可以维护启用开关、接管模式、OpenAI 兼容模型地址、模型名、API key、温度、超时、自动回复分类、置信度阈值、每日上限、单工单上限、冷却时间、AI 身份标识和自定义系统提示词。API key 保存后会加密存储，页面不会回显；留空保存时会保留已有密钥。

启用后，后台工单详情页可以生成 AI 回复草稿。草稿只会填入回复框，不会自动发送；管理员必须人工确认后再提交。

扩展也提供受控接管回复接口。该接口会重新生成回复、执行安全检查、执行敏感工单强制人工规则，并按扩展配置限制每日发送量、单工单发送量和冷却时间。它只会写入一条客服侧消息，不会修改工单状态。

当扩展配置为 `auto` 模式时，服务端调度器每 2 分钟扫描一次官方/系统工单。它只处理最新消息来自用户、非紧急、非 abuse、分类在 `autoReplyCategories` 内且未触发人工接管规则的工单；第三方托管节点工单不会被自动接管。

模板依赖两个受控接口：

```text
POST /api/tickets/:id/ai/context
POST /api/tickets/:id/ai/draft
POST /api/tickets/:id/ai/reply
```

这些接口只允许管理员访问，必须启用 `com.payincus.ai-ticket-agent` 扩展并授予对应权限。上下文只读取当前工单所属用户的脱敏摘要，不返回管理员内部备注、支付回调、密钥、root 密码或其他用户数据。

扩展设置页还会读取一个安全状态接口：

```text
GET /api/tickets/ai/status
```

该接口只返回扩展是否启用、当前模式、模型是否已配置、权限状态、自动分类、置信度阈值和限流配置等运营摘要，不返回模型地址、API key、后台路径、原始工单内容或用户数据。

## 安全边界

- 只有超级管理员可以安装、启用、禁用和卸载扩展。
- 用户端扩展不能访问 `/api/admin/*`。
- 扩展包不能包含绝对路径、`../` 路径、软链接或硬链接。
- 市场扩展必须来自 GitHub Release artifact，并校验 SHA256。
- 未上架市场扩展不会显示在默认市场，也不能直接安装。
- 安装市场扩展时会校验 PayIncus 版本兼容范围。
- 当前版本不执行扩展自带 shell 脚本，也不加载任意后端代码。
- 完整后端功能必须通过扩展 action runtime、API token/scope、事件系统和扩展存储等受控接口实现，不能绕过 PayIncus 的支付、余额、资源交付、权限和审计链路。
- 主题包已经通过独立 manifest、资产校验、主题市场安装、主题投稿审核/扫描/发布、预览、启用/回滚和配置表单机制接入；深层模板覆盖继续按受控边界补齐。
- AI 工单接管回复必须通过独立 `ticket:ai:reply` 权限，并会对退款、争议、账号安全、风控、删除/重装/迁移实例、凭据或后台细节、交付异常强制转人工。
- AI 工单接口不能调账、删除实例、修改工单状态或执行其他资源操作。
- AI 自动回复只在 `auto` 模式运行，并受每日上限、单工单上限和冷却时间约束。
